## 4. 开发经验与思路

### 4.1 从用户动作倒推技术，而不是从工具出发

稳定需求始终是“同事完成网页后，只说一句更新内网页面”。SFTP、SMB、Nginx 和 Docker 都只是实现手段。Windows 原生 SMB 胜出，不是因为它功能最多，而是因为它让真实用户少安装一个工具、少理解一层凭据。

### 4.2 便利不能替代服务器强制隔离

交接文件只告诉 Codex 应该访问哪个站点；真正的安全边界必须由 Samba `valid users`、独立账号和独立 Docker 卷强制执行。不能依赖提示词要求“不要访问其他目录”。

### 4.3 一个写入入口，每站独立读取面

第二个网站不应再创建一个争抢 `445` 的 Samba 容器。统一 Samba 降低端口和运维复杂度，每站独立 Nginx、卷和账号仍保留故障与权限边界。这是当前少量静态站点规模下的最小充分结构。

### 4.4 运行事实优先于历史文档

容器名、挂载、交接路径和服务状态会演进。更新文档前必须检查目标主机；公开文档则只保留匿名结构和验证结论。其他知识库只保留入口链接，避免两个“当前文档”逐渐冲突。

### 4.5 缓存问题必须分层取证

文件已更新而浏览器仍显示旧页面时，应分别检查服务器文件、`Last-Modified`、`ETag`、`Cache-Control`、HTTP 304、Service Worker 和强制刷新结果。不能把强制刷新有效直接解释为上传失败，也不应为了省事永久使用 `no-store`。

### 4.6 “命令执行完成”不等于“发布完成”

`robocopy` 成功只能证明同步命令结束。交接规则还需要检查静态产物、排除秘密文件、最后写入 `.internal/deployment.json`、从服务器回读完成标记并访问版本化预览地址。发布结果必须由服务端可观察事实确认。

### 4.7 运维接口默认安全，危险动作显式升级

日常验收必须只读；备份只短暂停止写入并尽力恢复；恢复默认只校验，只有明确提供确认词才允许清空卷。恢复包除校验和外，还必须拒绝路径穿越、链接和超出允许目录的归档内容。

### 4.8 备份文件不是恢复能力

只有在隔离环境把配置和网站卷恢复出来，再验证权限、HTTP 和真实发布，才能声称具备恢复能力。恢复时间、失败位置和人工处置方式也属于验收结果。

### 4.9 当前规模不需要平台化

少量静态站点、一人一站时，统一 Samba、独立卷和只读 Nginx 已经足够。没有出现多人协作、审批审计、版本回滚、动态后端或大量站点之前，不应引入内部 PaaS、第二套控制面或复杂发布 API。

### 完成情况

- [x] 已完成：部署拓扑、缓存策略、交接规则和安全运维脚本形成公开参考项目。
- [ ] 未完成：site-b 盲验、整机重启和隔离恢复；原因是这些需要对应终端、维护窗口或备用环境，不能用文档和本地脚本代替。

### 发现的问题

| 严重程度 | 问题描述 | 根本原因 | 改进建议 |
| --- | --- | --- | --- |
| 必须改正 | 早期文档曾把服务器自动检查接近描述为完整验收 | 没有严格区分自动检查、外部 Windows 和真实用户路径 | 继续使用五层验收模型，每层只记录自身证据 |
| 应当改正 | 历史阶段出现过运行路径、容器名与文档不一致 | 先编辑文档，后验证实时主机 | 所有运行事实修改前先确认主机身份并只读检查目标环境 |
| 应当改正 | 缓存问题初期容易被误判为 SMB 上传失败 | 没有先分离文件、HTTP 和浏览器三层证据 | 保留 ADR-004 的分层检查顺序和版本化预览 |
| 建议改进 | 完整复盘持续增长，查找当前状态的成本上升 | 同一文档同时承担历史、经验和待办 | 使用本阶段收口记录作为新入口，完整复盘继续承担历史证据链 |

### 做得好的地方

- 用户路径最终经过真实 Windows Codex 验证，而不是停留在设计推断；
- 权限隔离由服务器强制实施，没有把安全寄托于提示词；
- 公开仓库与真实凭据、网站内容和备份保持分离；
- 发现主机绑定卷与 Samba 删除语义不兼容后，改用命名卷并完成 CRUD 回归；
- 没有因为缓存问题或第二个站点而立即扩张为复杂平台。

### 下次重点关注

- 第一优先级是备份与隔离恢复闭环，不是新增发布功能；
- 有可靠备份后再进行 Docker Desktop 和 Mac mini 重启验收；
- site-b 盲验继续保持独立证据，不能从 site-a 结果外推；
- 如无法连接目标主机，只能记录“待实时复核”，不能引用旧状态充当当前事实。
