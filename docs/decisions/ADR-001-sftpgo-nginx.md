# ADR-001：采用 SFTPGo + Nginx 管理内网静态页面

## 状态

已由 [ADR-002：采用 Docker Samba + Nginx 管理 site-a](./ADR-002-samba-nginx-first-site.md) 取代。SFTPGo PoC 数据卷和备份仅作回滚保留。

## 日期

<VALIDATION_DATE>

## 背景

最初目标是让同事完成本地静态网页后，通过腾讯云 EdgeOne Makers 发布并获得访问地址。历史上已经验证过：

- Makers API Token 可以查询和部署 Direct Upload 项目；
- CLI 可以创建或更新项目；
- 自定义域名和临时鉴权地址可以用于预览；
- Git 仓库驱动项目与 Direct Upload 项目有不同的发布边界。

随后组织明确要求静态页面不得发布在公网，原公网方案失去合规前提。新约束为：

- 页面只能在组织内网访问；
- 使用 Mac mini 作为内网站点宿主；
- 多名同事可能分别维护不同静态网站；
- 当前阶段按“一人一个网站”设计；
- 同事主要使用 Codex，技术基础弱；
- 同事的操作应收敛为一句“更新内网页面”；
- IT 可以承担服务器、网络、账号和权限配置；
- 当前核心需求只是限制上传身份和网站目录，不需要完整 PaaS。

## 决策

使用以下组合：

- **SFTPGo**：负责 SSH 公钥认证、网站账号、主目录隔离、上传权限、配额和日志；
- **Nginx**：只读挂载网站目录，负责向组织内网提供静态页面；
- **项目侧固定部署脚本**：后续负责本地检查、同步、线上验证和结果反馈；
- **Codex 规则文件**：后续把“更新内网页面”映射到固定部署脚本。

当前稳定资源模型为：

```text
一名同事
→ 一个网站 ID
→ 一个 SFTPGo 账号
→ 一把独立 SSH 私钥
→ 一个受限网站目录
→ 一个固定内网访问地址
```

账号按网站命名，例如：

```text
site-a-writer
site-sales-demo
site-product-docs
```

端口属于网站，不属于人员。PoC 阶段可以按网站分配端口；正式环境优先使用内部 DNS 和统一的 80/443 入口。

## 当前 PoC 实现

```text
上传：<OLD_MAC_IP>:2022
账号：site-a-writer
页面：http://<OLD_MAC_IP>:8510/
管理：http://127.0.0.1:18080/（仅目标机本地）
```

SFTPGo 用户主目录为：

```text
/srv/sftpgo/site-a
```

Nginx 以只读方式读取相同 Docker 卷中的 `site-a` 目录。

## 选择 SFTPGo 的原因

- 原生支持 ARM64，已在 Apple M4 上验证为 `linux/arm64`；
- 当前仍在维护，2.7.1 包含路径越界和权限绕过修复；
- 不向用户提供普通 Shell，只提供文件传输能力；
- 支持仅公钥认证、主目录隔离、目录权限和配额；
- 可以把管理端限制在目标机回环地址；
- 比自行维护 OpenSSH chroot、forced-command 和权限脚本更不容易误配；
- 对“一人一个网站”而言，管理复杂度可控。

## 替代方案

### EdgeOne Makers

优点：部署体验成熟、CLI 简单、自动生成访问地址。

拒绝原因：组织禁止公网发布，产品边界与当前合规要求冲突。历史 Makers 项目和 Token 应作为待下线资产处理，而不是继续扩展。

### Gitee / Git 驱动发布

优点：有提交历史和自动部署。

拒绝原因：当前同事只需发布本地静态产物，不希望引入 Git 仓库和分支操作；同时 Git 不解决内网文件传输和宿主权限边界。

### CapRover

优点：支持本地 tar/CLI、App Token、自动域名和回滚；适合内网版 PaaS。

暂不采用原因：当前只需管理静态文件上传权限；CapRover 使用 Docker Swarm、Docker Socket，并需要 80/443/3000 等入口，安装在现有共享 M4 上的影响面过大。新 Mac mini 若未来扩展到动态应用，可重新评估。

### Coolify / Dokploy

优点：团队、权限、API 和应用管理完整。

暂不采用原因：更偏 Git、镜像或完整应用部署；正式安装更适合 Linux 主机/虚拟机，超出当前纯静态文件上传的需要。

### atmoz/sftp

优点：容器简单。

拒绝原因：官方 ARM64 支持问题长期未解决，不适合作为 Apple Silicon Mac mini 的新基线。

### macOS OpenSSH + 自定义限制脚本

优点：依赖少。

暂不采用原因：需要自行维护 chroot、forced-command、用户权限、日志和升级安全，误配后可能扩大到主机 Shell 权限。

## 后果

### 正面后果

- 同事不需要接触服务器管理界面；
- 每个账号只能访问自己的站点目录；
- 错误私钥不能登录；
- SFTP 账号不能执行普通 SSH 命令；
- 网页服务与上传服务职责分离；
- 方案可以随 Docker 卷和配置迁移到新 Mac mini。

### 代价与风险

- IT 需要负责账号、公钥、目录、端口/域名和生命周期；
- 当前单文件上传是原子的，但整站多文件更新仍可能出现短暂的新旧资源混合；
- 当前还没有自动回滚和版本保留；
- SFTPGo Community 使用 AGPL-3.0-only 并带附加条款，正式组织采用前需要完成内部许可确认；
- Docker Desktop 和主机重启后的无人值守恢复仍需纳入正式方案。

## 不变量

- 不得把私钥、密码、Makers Token 或 SFTPGo 管理凭据写入项目仓库；
- 不得向同事开放 SFTPGo 管理端；
- 不得给静态网站账号普通 SSH Shell；
- 不得让部署脚本接受任意主机、账号或远端目录；
- 不得把当前 PoC 地址描述为长期正式地址；
- 不得在组织禁止公网的前提下恢复 EdgeOne Makers 发布。

## 重新评估条件

出现以下任一条件时重新评估 CapRover、Coolify 或内部部署 API：

- 需要托管动态后端、数据库或多容器应用；
- 网站数量增长到手工账号/端口管理不可接受；
- 需要团队共享、审批、完整审计或自助开通；
- 需要正式内部 DNS、TLS、版本回滚和多节点高可用；
- 需要同一个网站由多人维护。
