# ADR-004：静态页面协商缓存与部署版本标识

- 状态：已采用
- 日期：<VALIDATION_DATE>
- 适用范围：site-a、site-b 及后续同类内网静态站点

## 背景

同事更新页面后，浏览器普通访问仍可能显示旧内容，强制刷新后才看到新版。现场检查确认：

- 服务器中文件已更新，页面没有 Service Worker；
- Nginx 默认返回 `ETag` 和 `Last-Modified`，但未明确返回 `Cache-Control`；
- 日志中存在较多 `304 Not Modified`，且可用条件请求复现；
- site-a 当时的首页约 13.4 MB，全面禁止缓存会带来明显的重复下载。

## 决策

### 1. 使用协商缓存

Nginx 对所有响应增加：

```nginx
add_header Cache-Control "private, no-cache" always;
```

保留 Nginx 默认的 `ETag` 和 `Last-Modified`。`no-cache` 不表示不允许保存，而是表示浏览器使用已保存内容前必须先向服务器校验：

- 文件未变：服务器返回 `304`，浏览器复用本地文件；
- 文件已变：服务器返回 `200` 和新内容。

### 2. 每次部署刷新校验标识

Codex 不直接从项目根目录或原构建产物同步，而是：

1. 将静态产物复制到独立的系统临时发布目录；
2. 生成唯一部署 ID，格式为 `yyyyMMddTHHmmssfffZ`；
3. 将临时发布目录中所有文件的 `LastWriteTimeUtc` 统一设为本次部署时间；
4. 再通过 `robocopy /MIR` 同步。

这可避免 Windows 同步保留旧时间戳时，新内容仍沿用旧 `Last-Modified` 或 `ETag` 的风险。临时目录保护了同事的源文件和原构建产物，不因发布改写它们。

### 3. 返回版本化预览地址

部署完成后，Codex 优先返回：

```text
http://<内网主机>:<站点端口>/?v=<deploymentId>
```

查询参数不改变静态文件内容，但会形成新的浏览器请求键，适合部署后立即验收。同时仍返回固定地址，便于日常收藏。

### 4. 保留部署完成标记

`.internal` 为发布系统保留目录，项目不得使用。主文件同步成功后，Codex 最后写入：

```text
.internal/deployment.json
```

文件只记录 `deploymentId` 和 `deployedAt`，不包含账号、密码或本地项目路径。只有标记写入成功，本次部署才算完整。

Codex 还必须通过 HTTP 回读该文件，确认其 `deploymentId` 与本次一致后才能报告成功。临时发布目录不得含有符号链接、junction 或其他 reparse point，避免将目录外文件意外上传。

## 未采用的方案

### `no-store` 或全部禁止缓存

实现简单，但会让大型 HTML 和静态资源每次全量下载，不符合内网快速验证的实际需求。

### 服务器推测“更新后两小时禁用缓存”

该策略无法追溯修改浏览器已经保存的旧响应，而且需要持续计算整站更新状态，复杂度高于协商缓存，收益更小。

### 在任意 HTML 中自动注入“网页已更新”脚本

它可以提供更强的用户提示，但要求改写同事提交的任意 HTML，并引入轮询、注入位置、CSP 和脚本兼容性问题。当前不实施；若未来页面有统一应用壳，可再在应用层实现。

## 结果与边界

- 页面未变时，只有小型校验请求，不会重复下载整页；
- 页面变更后，浏览器应通过普通访问校验并获取新内容；
- 部署人使用版本化地址可立即验收本次部署；
- 版本化参数只直接更换首页请求键；首页引用的 CSS、JavaScript 和图片仍依靠各自的协商缓存。若项目已支持内容哈希文件名，应继续使用；
- 在新策略生效前已被浏览器缓存的页面，可能仍需要最后一次强制刷新；它收到新响应头后，后续访问进入协商缓存流程；
- 浏览器的前进/后退页面快照不一定发起网络请求，不在本次 Nginx 策略的绝对控制范围内。
