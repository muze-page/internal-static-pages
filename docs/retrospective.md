# 内网静态页面管理：完整复盘、开发经验与后续计划

- 最后更新：<VALIDATION_DATE>
- 适用范围：当前 Mac mini 验证环境、site-a、site-b、新机重建与公开参考项目维护

## 文档定位

本文归纳从腾讯云 EdgeOne Makers 公网发布试验，到 Mac mini 内网托管、SFTPGo PoC、Docker Samba 多站点，再到协商缓存与新机全新部署的完整演进。

本文用于回答四个问题：

1. 为什么最终选择当前架构；
2. 同事和 Codex 应怎样发布网页；
3. IT 新增站点时应遵守哪些边界和验证步骤；
4. 缓存、部署版本和新 Mac mini 重建应怎样处理；
5. 如何把内部实践整理成不泄露环境信息的公开参考项目。

实时状态以目标 Mac mini 的容器、端口和验证结果为最高事实来源；具体决策与验证证据分别见 ADR 和站点部署验证记录。

### 关键演进概览

| 阶段 | 解决的问题 | 结论 |
|---|---|---|
| EdgeOne Makers 探索 | 验证 Codex 能否完成一句话公网发布 | 体验成立，但公网不符合后续合规要求 |
| SFTPGo PoC | 不开放宿主机 Shell 的账号与目录隔离 | 服务端可行，但 Windows 凭据与工具体验过重 |
| Docker Samba + Nginx | 让 Windows Codex 使用系统自带能力上传 | 成为当前方案 |
| 单 Samba 多站点 | 在同一 `445` 端口承载 site-a 与 site-b | 每站点独立身份、共享、卷和 HTTP 端口 |
| 协商缓存 | 避免更新后普通访问继续显示旧内容 | 使用 `private, no-cache` + 校验标识，不全面禁止缓存 |
| 新 Mac mini | 从临时验证机转向新设备 | 在本地项目是权威来源的前提下，重建空卷并由同事重新发布 |
| 独立公开参考项目 | 让方案可以单独维护和复用 | 只保留脱敏模板、决策、手册和检查脚本，运行数据继续留在内网 |
| 试运行与恢复闭环 | 避免把“配置写完”等同于“生产可用” | 增加只读验收、受保护备份、确认式恢复和分阶段签字 |

## 一、需求如何逐步收敛

### 1. 最初目标：让 Codex 直接发布公网静态网页

最初围绕腾讯云 EdgeOne Makers 的 `site-a` 项目验证了以下能力：

- 使用 Makers API Token 鉴权；
- 创建或更新静态页面；
- 使用项目域名预览；
- 省略 Gitee，让 Codex 直接部署本地静态产物；
- 把同事的操作收敛成一句“更新”。

这一步验证了核心产品体验：非技术同事不需要理解部署细节，Codex 可以完成“构建、上传、验证、返回地址”的闭环。

### 2. 合规条件改变：公网方案不再可用

组织随后明确要求页面不得发布到公网。问题因此不再是“怎样更方便地调用 Makers”，而是：

```text
怎样在组织内网保留 Makers 式的一句话发布体验
```

从这一刻起，Makers、Gitee 和公网域名退出当前发布链路。已在聊天或测试中使用过的 Makers Token 应撤销或轮换，不得复制到新文档、仓库或内网方案中。

### 3. 运行环境：先用现有 M4 验证，再迁移正式设备

组织已有一台内网 Mac mini M4，可安装 Docker，地址为 `<OLD_MAC_IP>`。该设备只承担快速验证，后续会迁移到新的 Mac mini，因此当前设计必须满足：

- 配置和数据可备份、可恢复；
- 网站之间相互隔离；
- 不依赖现有设备上的手工隐式状态；
- 地址变化时只需由 IT 更新交接配置；
- 不把临时 IP 描述为长期正式地址。

### 4. 使用模型：一人一个静态网站

需求进一步收敛为：

- 同事主要部署静态 HTML、CSS、JavaScript、图片和字体；
- 当前一名同事维护一个网站；
- 多名同事可以分别维护不同网站；
- IT 负责建站、账号、密码、共享目录、端口和备份；
- 同事不需要学习 Docker、SSH、SFTP、服务器目录或端口分配；
- 同事只需在项目 Codex 中说“更新内网页面”。

这使“上传权限隔离”成为主要矛盾，暂时不需要建设完整 PaaS、自助管理后台或审批平台。

## 二、方案演进与取舍

### 阶段 A：EdgeOne Makers

优点是发布体验成熟、API 简单、自动提供预览地址。被放弃的原因不是技术不可行，而是公网托管与组织合规边界冲突。

经验是：技术验证成功不等于方案仍然有效；合规前提改变后，必须停止沿旧路径继续优化。

### 阶段 B：Mac mini + SFTPGo + Nginx

为避免向同事开放宿主机 SSH，曾在 Docker 中部署 SFTPGo：

- SFTPGo 管理账号、密钥、目录隔离和上传；
- Nginx 只读挂载网站数据并提供内网 HTTP；
- 每个网站使用独立账号、目录和访问端口；
- 从另一台 Mac 完成了上传、错误密钥拒绝、Shell 禁止、持久化和 HTTP 验证。

该 PoC 证明了权限模型可行，但真实使用者是 Windows 用户。Windows 自带 `sftp.exe` 不适合从项目 `.env` 中稳定、非交互地传递密码；使用密钥又增加生成、安装、路径和权限管理；安装 WinSCP 则增加同事操作和终端差异。

因此 SFTPGo 是有效的技术验证，但不是当前“Windows + Codex + 极少操作”场景的最简答案。

### 阶段 C：Docker Samba + Nginx

Windows 原生支持 SMB，PowerShell、`net use` 和 `robocopy` 均为系统自带能力。最终选择：

- Docker Samba 负责账号认证和共享目录隔离；
- 每个网站使用独立 Docker 卷；
- 每个网站使用独立 SMB 账号和随机密码；
- Nginx 只读挂载对应网站卷；
- 项目根目录只需放置 `.env` 与 `AGENTS.md`；
- Codex 使用 Windows 原生工具同步并验证页面。

没有启用 macOS 系统级文件共享，避免把宿主机已有公共文件夹或其他目录带入暴露面。

### 阶段 D：一个 Samba 容器承载多个隔离站点

新增 `site-b` 时确认：同一 IP 的 SMB 标准端口 `445` 只能由一个服务监听。因此不能按网站启动多个都绑定 `445` 的 Samba 容器。

当前模型调整为：

```text
一个 Samba 容器、一个 445 端口
├── site-a：独立共享、账号、密码、Docker 卷、网页端口 8520
└── site-b：独立共享、账号、密码、Docker 卷、网页端口 8530
```

Samba 通过每个共享的 `valid users`、`force user` 和 `force group` 实现隔离；每个 Nginx 容器只读挂载一个网站卷。

### 阶段 E：协商缓存与可验证部署

实际更新中出现过“服务器文件已更新，普通访问仍看到旧内容，强制刷新才正常”。检查确认页面没有 Service Worker，Nginx 已有 `ETag` 和 `Last-Modified`，但未明确要求浏览器重新校验。

最终没有使用每次全量下载的 `no-store`，也没有设计“更新后两小时禁用缓存”的服务器状态机。现行服务端策略与交接规则是：

- Nginx 返回 `Cache-Control: private, no-cache`，保留 `ETag` 和 `Last-Modified`；
- 未变文件返回 `304`，变更文件返回 `200`；
- Codex 在独立临时目录组织产物，每次生成唯一部署 ID；
- 主同步后最后写入 `.internal/deployment.json`，并通过 HTTP 回读核对；
- 成功后返回 `?v=<deploymentId>` 预览地址和固定地址。

这一阶段的重点不是“绝对不缓存”，而是“保留复用，但使用前必须校验；部署完成必须可验证”。

### 阶段 F：拆分为独立公开参考项目

方案稳定后，将原先散落在内部文档仓库中的架构决策、脱敏配置、交接模板和验证记录整理为独立的 `internal-static-pages` 项目。拆分遵循以下边界：

- 独立项目是当前方案的唯一文档与代码事实源；
- 原内部文档仓库只保留项目索引，不再保存重复副本；
- 真实 `.env`、`users.conf`、网站内容、备份、内网地址、域名、人员和交接路径不得进入公开仓库；
- `site-a`、`site-b` 和尖括号占位符只表达拓扑，不对应真实人员或环境；
- GitHub 只用于发布参考项目，不启用 GitHub Pages，也不承载实际网页。

公开前完成了 Git 历史、示例文件、绝对路径、秘密和身份标识检查，并为项目选择 Apache License 2.0。这里的“公开”只是让通用方案可复用，不改变“实际页面只能在组织内网访问”的产品边界。

### 阶段 G：把试运行、备份和恢复变成可执行接口

新机手册解决“怎样部署”，但不能单独证明“已经可交付”。因此进一步增加：

- `acceptance-check.sh`：只读检查 Docker、Compose、容器、Samba `valid users`、HTTP 状态和安全/缓存响应头；
- `backup.sh`：短暂停止 Samba 写入，保持 Nginx 只读访问，并备份运行配置和两个网站卷；
- `restore.sh`：默认只校验备份，只有显式提供 `--apply RESTORE` 才执行破坏性恢复；
- `test-operation-contracts.sh`：验证错误参数、越界路径、符号链接和错误确认词会被拒绝；
- 试运行验收清单：把旁路部署、Windows 权限矩阵、Codex 盲操作、重启和恢复演练分开签字。

这些工具缩小了操作失误空间，但不把本地脚本通过包装成真实新机验收完成。Windows 凭据隔离、CRUD、真实 Codex 发布、Docker Desktop 登录后恢复和备用环境恢复仍必须在目标环境执行。

## 三、当前实际架构

截至 <VALIDATION_DATE>，临时验证机上的现行资源为：

| 网站 | SMB 共享 | SMB 账号 | 页面地址 | 数据卷 |
|---|---|---|---|---|
| site-a | `\\<OLD_MAC_IP>\site-a` | `site-a-writer` | `http://<OLD_MAC_IP>:8520/` | `internal-pages-site-a-data` |
| site-b | `\\<OLD_MAC_IP>\site-b` | `site-b-writer` | `http://<OLD_MAC_IP>:8530/` | `internal-pages-site-b-data` |

统一 Samba 容器为 `internal-pages-samba`，监听内网 `445`。两个账号只能访问各自共享，错误密码、访客访问和交叉访问均被拒绝。

两个 Nginx 站点已启用 `Cache-Control: private, no-cache`。现场验证为：普通请求 `200`，携带当前 `ETag` 的条件请求 `304`，携带过期校验值的请求 `200`。

### 资源占用快照

<VALIDATION_DATE> 空闲时现场测量，三个容器 CPU 均接近 `0%`：

| 容器 | 内存 |
|---|---:|
| Samba | 约 19.6 MiB |
| site-a Nginx | 约 4.7 MiB |
| site-b Nginx | 约 4.7 MiB |

容器合计约 29 MiB。当时 site-a 数据卷约 25.6 MiB，site-b 数据卷约 8 KiB；Samba 和 Nginx 镜像逻辑大小约为 31.5 MB 和 25.8 MB。这是某一时点的负载快照，不包含 Docker Desktop 虚拟机本身的常驻开销，也不代表并发访问时的峰值。

### 用户看到的流程

```text
本地完成网页
→ 对 Codex 说“更新内网页面”
→ 等待 Codex 返回本次版本化预览地址和固定地址
```

### Codex 内部执行的流程

```text
识别项目与构建方式
→ 复制到独立系统临时发布目录
→ 检查 index.html、敏感文件与 reparse point
→ 生成部署 ID 并更新发布文件时间
→ 脱敏读取 .env
→ 临时连接固定 SMB 共享
→ robocopy 镜像同步
→ 最后写入并回读 deployment.json
→ 断开临时连接
→ 验证版本化首页、缓存头和主要资源
→ 返回版本化地址、固定地址或脱敏错误
```

项目在 Windows 上的绝对路径不是部署配置。只要 `.env` 和 `AGENTS.md` 位于项目根目录，项目移动后仍可使用。

## 四、核心开发与运维经验

### 1. 先明确边界，再比较工具

早期工具候选包括 Makers、Gitee、SFTPGo、原生 SSH、Samba、CapRover、Coolify 和 Dokploy。真正决定选型的不是功能多少，而是以下边界：

- 只能在组织内网访问；
- 当前只托管静态文件；
- 一人一个网站；
- 同事使用 Windows 和 Codex；
- IT 可以承担初始配置；
- 用户操作必须尽量少。

边界明确后，完整 PaaS 的团队、Git、镜像、数据库和多应用能力反而是额外负担。

### 2. 把用户体验与技术实现分层

同事只需要记住一句“更新内网页面”。构建、凭据、共享连接、同步、端口和健康检查都属于 Codex 与 IT 的内部实现。

这不是隐藏必要知识，而是把稳定决策固化到 `.env` 和 `AGENTS.md`，避免每次发布重新做技术选择。

### 3. 端口和账号归属网站，不归属人员

网页端口、共享名和数据卷都是网站属性。人员变更时可以轮换账号和密码，而网站地址不必变化。

SMB 的 `445` 是统一传输入口，不需要给每位同事分配不同 SMB 端口；网页预览端口则按网站独立分配。未来使用内部 DNS 后，可把端口隐藏在统一的 80/443 反向代理后。

### 4. 一站点一身份、一共享、一数据卷

当前即使是一人一个网站，也不能让所有同事共用一个写入账号或公共目录。稳定的不变量是：

```text
一个网站 ID
→ 一个 SMB 共享
→ 一个独立账号和密码
→ 一个独立 Docker 数据卷
→ 一个只读网页服务
→ 一个固定预览地址
```

这样可以分别撤销、轮换、备份和迁移，也能用交叉访问测试证明隔离有效。

### 5. 优先使用同事系统已有能力

SFTPGo 的权限能力比 Samba 更丰富，但 Windows 同事需要额外处理 SFTP 密码、SSH 密钥或第三方客户端。SMB、PowerShell 和 `robocopy` 已包含在 Windows 中，更符合“让 Codex 做、同事少操作”的目标。

工具选择应围绕真实客户端，而不是只看服务器侧是否优雅。

### 6. 密码可以放项目 `.env`，但必须接受明确约束

当前用户已选择密码方案，并允许将密码放入项目根目录 `.env`。这是一种以易用性换取本地秘密管理责任的取舍，必须配套：

- 每个网站使用不同随机密码；
- `.env` 权限尽可能收紧；
- `.env` 必须加入 `.gitignore`；
- Codex 不得打印密码或包含密码的命令；
- `.env` 不得进入发布目录、Git、文档、聊天或日志；
- 密码泄露时只轮换对应站点，不影响其他网站；
- 交接文件只发给对应同事。

项目绝对路径无需预先知道，也不应写进统一模板。

### 7. 规则文件必须固定目标，不能让部署临场发挥

站点 `AGENTS.md` 应明确：

- 只能读取本项目 `.env` 中的固定目标；
- 不得更换服务器、共享、账号或预览地址；
- 不得在失败后改用公网、其他账号或其他上传工具；
- 失败时停止并提醒联系管理员；
- 成功后返回本次版本化预览地址和固定内网链接。

“提醒不要传错”不是权限边界；固定配置、服务器隔离和默认拒绝才是。

### 8. `robocopy /MIR` 前必须先确认发布源

镜像同步会删除远端不存在于本地源目录中的文件。如果 Codex 错把项目根目录或空目录当作发布源，可能清空网站或上传 `.env`、源码缓存与日志。

同步前必须：

1. 运行项目现有构建命令；
2. 优先选择明确的 `dist`、`build` 或 `out`；
3. 无论是构建产物还是纯 HTML，都复制到独立的系统临时发布目录；
4. 确认存在 `index.html`；
5. 扫描 `.env`、私钥、`.git`、`.internal`、日志、符号链接、junction 和其他 reparse point；
6. 生成部署 ID，再执行镜像同步；
7. 主同步后最后写入完成标记，并回读验证。

`robocopy` 返回码 `0`–`7` 是成功或非致命差异，`8` 及以上才视为失败。

### 9. Docker Desktop 的宿主机绑定目录不等于 Linux 文件系统

首次 Samba 实现直接绑定 macOS 宿主机目录，上传和重命名成功，但删除返回 `NT_STATUS_NOT_SUPPORTED`。改用 Docker 命名卷后，创建、读取、重命名和删除全部恢复正常。

经验是：文件共享协议依赖的锁、权限、重命名和删除语义必须实测，不能只验证“能上传一个文件”。

### 10. 标准端口决定服务拓扑

同一 IP 上只能有一个服务占用 SMB `445`。新增站点时不应复制一个新的 Samba 容器，而应在统一 Samba 配置中增加用户、共享和卷。

网页 Nginx 可以按站点独立，因为每个站点使用不同 HTTP 端口；传输入口和页面入口需要分开建模。

### 11. 变更共享基础设施时先旁路验证

新增 site-b 时，先备份 site-a，再在备用端口启动临时 Samba 配置，验证双用户、双共享和交叉拒绝，最后才切换正式 `445`。

共享基础设施的安全变更顺序应为：

```text
备份
→ 创建新卷与账号
→ 备用端口验证新配置
→ 正式切换
→ 外部客户端验证
→ 重启验证
→ 清理临时服务
```

这比直接修改正式容器更容易发现配置错误，也保留明确回滚点。

### 12. 验证必须同时包含成功路径和拒绝路径

一个站点“能打开”不足以证明部署正确。当前采用的验证矩阵包括：

- 正确账号能登录自己的共享；
- 创建目录、上传、读取、重命名、删除成功；
- 错误密码被拒绝；
- 访客访问被拒绝；
- 同事 A 不能访问 site-b；
- 同事 B 不能访问 site-a；
- Nginx 对数据卷只读；
- 容器重启后数据与访问恢复；
- 从另一台设备访问 HTTP 返回 200；
- 新站点上线后旧站点保持正常；
- 备用端口和临时目录已清理。

负向测试是权限验证的一部分，不是可选补充。

### 13. 先确认操作主机，再读取运行状态

本地 Codex 所在的 Mac 与 `<OLD_MAC_IP>` 测试机是两台设备。检查 Docker、配置路径、端口和备份前，必须先确认当前主机；目标机检查应通过 SSH 明确执行。

“命令执行成功但执行在错误机器上”比命令报错更危险。

### 14. 历史方案、当前事实和未来计划必须分开

Makers、SFTPGo 和 Samba 都曾经真实可用，但只有 Samba 多站点方案代表当前事实。文档应始终区分：

- 历史上验证过什么；
- 当前实际运行什么；
- 哪些事项仍未完成；
- 哪些条件会触发重新选型。

旧 ADR 不删除，只标记被新 ADR 取代，以保留决策证据链。

### 15. `no-cache` 不等于不使用缓存

强制每次全量下载可以简化思考，但 site-a 当时单个首页约 13.4 MB，`no-store` 会把缓存问题变成带宽和等待问题。

`private, no-cache` 的实际语义是：浏览器可以保存响应，但复用前必须用 `ETag` 或 `Last-Modified` 与服务器校验。未变时只产生小型 `304` 响应，变更时才下载新内容。

“更新后两小时强制刷新”看似精细，但服务器无法追溯修改浏览器已有的旧缓存响应，还会引入全站更新窗口状态。对当前规模，持续协商校验更简单可靠。

### 16. 部署成功必须有可核对的事实

仅看 `robocopy` 退出码或首页 `200` 不能证明“这一次部署”已完整到达。现行流程使用三层证据：

1. `robocopy` 退出码 `0`–`7`；
2. 最后写入 `.internal/deployment.json`；
3. 通过 HTTP 回读并核对 `deploymentId`，再请求带同一 ID 的预览地址。

这不是整站原子发布，但它能把“同步命令已执行”提升为“服务器可读到本次完成标识”。

### 17. 不要自动改写无法统一管理的网页

“页面已更新，点击刷新”提示需要客户端脚本、轮询和可控的应用壳。当前同事可以上传任意静态 HTML，服务端注入会遇到脚本位置、CSP、编码和页面兼容性问题。

因此当前不注入更新提示，不改写同事的产物。若未来所有站点统一使用同一应用壳，再在应用层实现版本检测和提示。

### 18. 新机重建方式取决于内容权威来源

当 同事 A、同事 B 本地项目保存了网站全部最新内容时，新 Mac mini 不需要迁移旧 Docker 卷。可以在新机重建空卷、轮换密码，再让同事从项目重新发布。这比迁移临时验证机的运行数据更简单，也会顺带验证交接流程。

但如果旧网站卷存在本地项目没有的手工文件或唯一数据，就必须停止“空卷重建”，改用经过校验和恢复演练的卷迁移。迁移方式应由权威数据来源决定，不是由工具偏好决定。

### 19. 可复用方案与真实运行资产必须物理分离

把项目公开前，最重要的工作不是改写几处名称，而是重新划定资产边界：

```text
公开仓库
→ ADR、脱敏模板、Compose/Nginx/Samba 参考配置、检查脚本

内网环境
→ 真实密码、users.conf、同事交接包、网站内容、Docker 卷和备份
```

如果同一份文档既承担公开教程又记录真实主机事实，后续修改很容易重新泄露地址、身份或凭据。独立仓库建立后，内部文档仓库只保留链接，避免两个副本逐渐给出不同答案。

### 20. 公开仓库发布本身是一项安全发布

即使源码看起来只有 Markdown、YAML 和 Shell，也要按发布制品对待：

1. 检查当前差异和 Git 历史，而不是只看工作区文件；
2. 扫描 Token、密码、私钥、真实 IP、域名、人员名和个人绝对路径；
3. 用明显占位符替换环境值，避免“假示例”仍能关联真实组织；
4. 明确仓库公开不等于服务公网化，不启用 Pages；
5. 先选定许可证，再公开复用边界；
6. 推送后以远端提交和 CI 结果作为发布证据。

Apache License 2.0 适合本项目的原因是它允许组织和个人复用、修改及分发配置与脚本，同时保留版权、许可证声明和专利授权要求。GPL 的强 copyleft 要求与 AGPL 面向网络交互场景的源码提供义务都不是这个小型参考项目当前需要施加的约束。

### 21. 运维接口应默认安全，危险动作必须显式升级

脚本的命令行设计采用三个层次：

- 日常验收只读，不改变容器、卷或网页；
- 备份只暂停写入入口，读取页面继续服务，并在退出时尽力恢复 Samba；
- 恢复默认只校验，必须输入明确确认词才允许停止服务、清空卷和写回配置。

恢复包不能只验证 SHA-256。因为校验清单与归档可能同时被替换，恢复脚本还必须限制文件数量、归档条目类型和目标路径，拒绝绝对路径、`..`、符号链接、硬链接以及 `deploy/` 之外的运行配置。

这一经验可推广到其他内部运维工具：默认行为应可重复、只读或可逆；不可逆步骤需要明确升级，并在失败信息中说明系统可能停留在哪种状态。

### 22. 自动验收只能覆盖服务器事实，不能代替真实用户路径

自动脚本适合稳定检查：容器是否运行、配置能否解析、HTTP 是否返回 200、响应头是否正确、共享是否声明了预期账号。它无法证明：

- Windows 凭据缓存没有导致串号；
- 错误密码、访客和交叉共享一定被外部客户端拒绝；
- `robocopy` 能完成真实 CRUD 和镜像同步；
- 同事只说“更新内网页面”时，Codex 会选择正确构建产物；
- Mac 重启且管理员登录后，Docker Desktop 和全部服务能恢复。

因此验收必须分为“服务器自动检查”“另一台 Windows 权限矩阵”“真实 Codex 盲操作”“重启恢复”“隔离环境恢复演练”五层。没有对应证据的层级只能标记待验收。

### 23. 备份脚本存在不等于恢复能力成立

备份完成只证明文件被写出；恢复能力还需要：

- 校验归档完整性和路径安全；
- 在备用主机或隔离环境执行恢复；
- 验证配置权限、网站卷内容、Samba 账号隔离和 HTTP 页面；
- 再执行一次真实 Codex 发布；
- 记录恢复时间、失败处置和是否满足业务回退窗口。

当前恢复过程不是原子操作。失败时服务可能保持停止状态，因此不能在没有当前环境第二份安全备份、维护窗口和管理员在场的情况下直接演练。

## 五、新增静态网站的标准方法

新增一名同事和一个网站时，IT 应按以下顺序执行：

1. 生成不重复的网站 ID，例如 `<name>page`；
2. 检查并登记未占用的网页端口；
3. 创建独立 Docker 数据卷和默认 `index.html`；
4. 创建独立 `site-<site-id>` SMB 账号和随机密码；
5. 在统一 Samba 配置中添加共享和 `valid users` 限制；
6. 启动只读挂载该卷的 Nginx 服务，并统一配置协商缓存响应头；
7. 在备用端口验证新配置和所有旧站点回归；
8. 切换正式 Samba 后完成 CRUD、交叉拒绝、HTTP 和重启验证；
9. 在 IT 本机生成只含该站点信息的 `.env`、`AGENTS.md` 和简短说明；`AGENTS.md` 必须包含临时发布目录、部署 ID、完成标记回读和版本化预览规则；
10. 扫描交接包和文档，确保没有其他站点密码、Makers Token 或私钥；
11. 把 `.env` 与 `AGENTS.md` 复制到同事项目根目录；
12. 让同事在真实 Windows Codex 中执行一次“更新内网页面”验收。

不要为了新增站点而启用宿主机全局 SMB、复用他人账号、开放 SSH Shell，或增加第二个监听 `445` 的 Samba 容器。

## 六、当前风险与未完成事项

### 上线前必须补齐

1. site-a 真实 Windows Codex 盲操作已于 2026-07-17 得到结果级通过确认；site-b 仍需独立完成，且 site-a 的结果不能外推；
2. 撤销或轮换历史 Makers API Token，并确认旧公网入口已下线；
3. 使用现有备份工具建立定时备份，并在备用环境完成一次真实恢复演练；
4. 建立网站 ID、账号、端口、卷、负责人和交接状态的资产清单；
5. 为新 Mac mini 配置固定 IP 或 DHCP 保留、主机加固和无人值守重启恢复；
6. 验证 Docker Desktop 登录后启动、容器自动恢复和断电后的状态；
7. 制定 SMB 密码轮换、人员离职和站点下线流程。

### 当前已知技术限制

- `robocopy /MIR` 是多文件原地更新，可能短暂出现新旧 HTML、CSS 和 JavaScript 混合；
- 当前没有整站版本目录、原子切换和自动回滚；
- 部署 ID 和完成标记可以证明某次同步已完成，但不能消除多文件更新期间的短暂混合状态；
- `?v=<deploymentId>` 只直接更换首页请求键，CSS、JavaScript 和图片仍依靠各自的协商缓存；有构建能力的项目应继续使用内容哈希文件名；
- 新缓存策略生效前已被浏览器保存的旧页面，可能还需要最后一次强制刷新；
- 当前依赖 IP 与端口，不具备易记内部域名和内部 TLS；
- 密码明文存在同事项目 `.env`，安全性依赖本地电脑、Git 忽略和 Codex 规则；
- 当前是单台临时设备，不具备高可用；
- 网站增多后，手工维护账号、端口和 Compose 配置会逐渐变重。
- 当前恢复流程不是原子操作，中途失败时可能需要管理员手工恢复服务；
- 自动验收不能替代真实 Windows、Codex 和整机重启验证。

## 七、新 Mac mini 的全新部署顺序

### 阶段 1：确认内容权威来源

- 让 同事 A、同事 B 确认本地项目能构建，且包含网站所有 HTML、CSS、JavaScript、图片和字体；
- 检查旧站点是否有本地项目不存在的手工文件或唯一数据；
- 只有本地项目是完整权威来源时，才使用“新空卷 + 同事重新发布”；
- 若存在唯一服务器文件，停止本流程并改用卷迁移。

### 阶段 2：准备正式主机

- 安装并锁定 Docker 运行环境；
- 优先为新机配置新的固定内网地址或 DHCP 保留，不默认抢占旧机 `<OLD_MAC_IP>`；
- 限制防火墙，仅允许组织内网访问 SMB 和页面入口；
- 配置磁盘空间、备份目标、监控和自动启动；
- 明确 Docker Desktop 需要管理员登录后才能自动启动的运维边界，并实际演练重启；
- 不启用无关公网入口。

### 阶段 3：重建与旁路验收

- 在新主机创建全新空卷，不恢复旧网站卷；
- 重新生成 site-a 和 site-b 不同的随机密码；
- 启动统一 Samba 和各站点 Nginx，保持已验证的站点 ID、账号、共享、端口和容器拓扑；
- 使用临时 IP 或测试域名完成全套验证矩阵；
- 重新生成两个独立交接包，不带任何历史凭据；
- 完成密码错误、访客、交叉访问、CRUD、HTTP、缓存和重启验证。

### 阶段 4：同事重新发布与切换

- 先确认新机已使用最终 IP；
- IT 向 同事 A、同事 B 分别交付对应的新 `.env` 和 `AGENTS.md`；
- 两位同事各自只需说“更新内网页面”，将本地最新产物发布到新空卷；
- IT 验证占位页已替换、完成标记正确、资源无 404，且交叉访问仍被拒绝；
- 旧机保留 7–14 天回退窗口，但不再接受新发布；
- 新机完成首份配置和网站卷备份后，才考虑下线旧机。

如果新主机地址发生变化，只需由 IT 更新各项目 `.env` 中的共享与预览地址；同事无需知道项目绝对路径，也无需学习新的发布命令。只有旧机其他全部服务也已迁移或下线时，才应让新机接管 `<OLD_MAC_IP>`。

## 八、何时重新评估当前方案

出现以下任一情况时，应重新评估内部部署 API、反向代理、版本发布服务或内网 PaaS：

- 一个网站需要多人共同维护；
- 网站数量增长到手工账号、端口和配置管理不可接受；
- 需要审批、审计检索、自助开通或精细角色权限；
- 需要原子发布、版本回滚和发布历史；
- 需要统一内部域名、TLS 和单点登录；
- 开始托管后端、数据库或多容器应用；
- 需要多节点高可用。

在这些条件出现前，继续保持“一套统一 Samba、每站点独立身份和卷、每站点独立只读 Nginx、Codex 一句话发布”的小型方案，更符合当前问题规模。

## 十、最终结论

本项目最重要的成果不是选择了 Samba，而是形成了稳定的职责边界：

```text
同事负责网页内容
Codex 负责构建、同步和验证
IT 负责身份、目录、端口、备份和迁移
服务器负责强制隔离，不依赖口头约束
```

当前方案已在临时 M4 上证明两个独立网站可以共享一个 SMB 服务而保持目录隔离，协商缓存也已在两个 Nginx 站点上完成现场验证。部署版本标识与可回读完成标记已写入同事交接规则，公开参考仓库也已建立自动检查、备份和确认式恢复接口。site-a 的真实 Windows Codex 盲操作已于 2026-07-17 得到结果级通过确认；site-b、整机重启和隔离恢复仍需各自的实践证据。

该方案的合理边界仍是“少量、静态、一人一站”。本阶段在 site-a 盲验通过后停止扩张；下一阶段应优先完成首份受保护备份和隔离恢复闭环，再处理整机重启、site-b 盲验、历史 Token 下线和新 Mac mini 旁路迁移。
